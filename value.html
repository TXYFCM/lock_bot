僵尸锁自动识别

2 小时滑动窗口持续扫描：有 Lock Bot 锁但 XPU 与显存利用率双双低于 10% → 标记为僵尸锁。精准识别"锁而不用"的浪费，为资源回收提供事实依据。

资源占用可追溯

Lock Bot 锁记录 + 当天历史占用完整留存，谁在什么时间锁了哪张卡、锁了多久——全程可查。防止资源圈占，支撑公平调度。

⚡

运营效率提升

60 秒自动刷新 + 双数据源融合（Lock Bot 占用 + Monquery 利用率），替代人工逐节点巡检。异常状态（僵尸锁）主动标记，问题发现前置到影响业务之前。

💰

硬件成本优化

识别长期低利用率节点 → 评估是否释放或重新分配。单张 XPU 卡成本可观，提升 10% 利用率即相当于节省数十万硬件投入，ROI 清晰可量化。

数据 → 决策

监控指标如何直接指导管理动作——三个典型场景

🔼

场景一：扩容决策

触发信号：XPU 平均利用率连续多日 > 50%

当 average 视图显示集群 XPU 利用率持续处于高位（紫色曲线长期 > 50%），且 FREE 节点占比持续走低，说明现有资源已接近瓶颈。Leader 可据此数据向上申请新增节点预算，有据可依。

数据来源：average 视图 · XPU 移动平均曲线 · FREE 节点占比趋势
🔽

场景二：资源回收

触发信号：节点 LOCKED 但利用率 < 10% 持续 ≥ 2 小时（僵尸锁）

节点列表中红色"⚠已锁定≥2h未使用"标记出现时，说明该节点被 Lock Bot 锁定但实际未使用。管理者可直接联系持锁人确认是否仍需占用，释放后可供其他团队使用，避免 GPU 卡空转浪费。

数据来源：个人视图 · 僵尸锁 badge · 占用红条 hover 查看持锁人
⚙️

场景三：调度策略优化

触发信号：PARTIAL 状态节点占比持续偏高

PARTIAL 节点 = 8 卡中仅部分被占用（1~7 卡 BUSY）。若此类节点占比过高，说明整机锁（NODE 模式）造成大量碎片化浪费。Leader 可推动切换为单卡锁（DEVICE 模式），将空闲卡分配给其他任务，提升整体利用率。

数据来源：个人视图 · PARTIAL badge 统计 · 展开查看逐卡详情
